# Архитектура системы мониторинга учебного оборудования

## 1. Сущности и их связи

### Диаграмма связей

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐
│    User     │────<│  Classroom   │────<│ Workstation │
│  (Пользов.) │     │  (Кабинет)   │     │ (Раб.место) │
└─────────────┘     └──────────────┘     └─────────────┘
      │                                        │
      │                                        │
      ▼                                        ▼
┌─────────────┐                         ┌─────────────┐
│IssueReport  │────────────────────────>│  Equipment  │
│ (Обращение) │                         │(Оборудован.)│
└─────────────┘                         └─────────────┘
      │                                   │    │    │
      │                                   │    │    │
      ▼                                   ▼    ▼    ▼
┌─────────────┐                    ┌────────┐ ┌────┐ ┌───────────┐
│   Repair    │                    │Compone-│ │Soft│ │CustomField│
│  (Ремонт)   │                    │  nt    │ │ware│ │   Value   │
└─────────────┘                    └────────┘ └────┘ └───────────┘
```

### Описание сущностей

| Сущность | Описание | Связи |
|----------|----------|-------|
| **User** | Пользователи системы (админ/преподаватель) | → Classroom (ответственный), → IssueReport, → Repair, → Notification |
| **Classroom** | Учебные кабинеты | → User (ответственный), → Workstation[] |
| **Workstation** | Рабочие места в кабинетах | → Classroom, → Equipment[] |
| **Equipment** | Оборудование (ПК, мониторы, принтеры и т.д.) | → Workstation, → Component[], → Software[], → IssueReport[], → Repair[] |
| **Component** | Компоненты ПК (CPU, RAM, HDD и т.д.) | → Equipment |
| **Software** | Каталог программного обеспечения | → InstalledSoftware[] |
| **InstalledSoftware** | Установленное ПО на оборудовании | → Equipment, → Software |
| **IssueReport** | Обращения о неисправностях | → Equipment, → User, → Repair[] |
| **Repair** | Записи о ремонтах | → Equipment, → IssueReport, → User (исполнитель), → User (создатель) |
| **Notification** | Уведомления пользователей | → User |
| **ChangeHistory** | История изменений | → User, → Equipment (опционально) |
| **CustomFieldDefinition** | Определения пользовательских полей | → CustomFieldValue[] |
| **CustomFieldValue** | Значения пользовательских полей | → CustomFieldDefinition, → Equipment |

## 2. Статусы оборудования

```typescript
enum EquipmentStatus {
  OPERATIONAL       // Исправно - оборудование работает нормально
  NEEDS_CHECK       // Требует проверки - есть подозрения на неисправность
  IN_REPAIR         // В ремонте - оборудование находится на ремонте
  DECOMMISSIONED    // Списано - оборудование выведено из эксплуатации
  NOT_IN_USE        // Не используется - оборудование временно не используется
}
```

## 4. Слои архитектуры

### 4.1. Слой данных (Prisma)
- `prisma/schema.prisma` — схема базы данных
- `lib/db.ts` — singleton клиент Prisma

### 4.2. Сервисный слой
- `lib/services/*` — бизнес-логика
- Каждый сервис отвечает за одну сущность
- Методы: CRUD, фильтрация, пагинация, статистика

### 4.3. Слой валидации
- `lib/validators/index.ts` — Zod схемы
- Валидация на уровне API routes

### 4.4. API слой
- `app/api/*` — REST API endpoints
- `lib/api/response.ts` — унифицированные ответы

### 4.5. Слой типов
- `lib/types/index.ts` — TypeScript типы
- Re-export Prisma типов + DTO

## 5. Паттерны и практики

### Мягкое удаление
Все основные сущности поддерживают мягкое удаление через флаг `isActive`.

### Транзакции
Операции, затрагивающие несколько сущностей, выполняются в транзакциях (например, создание ремонта с изменением статуса оборудования).

### Пагинация
Все списковые методы поддерживают пагинацию с сортировкой.

### История изменений
Система логирует изменения в таблицу `change_history` для аудита.

### Уведомления
Автоматические уведомления при ключевых событиях (создание обращения, назначение ремонта и т.д.).
